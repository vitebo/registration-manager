# `.eslintrc.cjs`

Além das regras padrão de lint que vieram no boilerplate, adicionei algumas regras que considero interessantes para manter o código mais limpo e organizado.

Uma dessas regras refere-se aos `import` e `export` da aplicação. Entre elas, temos uma regra bastante rigorosa chamada [`import/no-internal-modules`](https://github.com/import-js/eslint-plugin-import/blob/main/docs/rules/no-internal-modules.md), que pode ser um pouco desafiadora caso você não leia a documentação sobre como ela funciona.

A grande vantagem dessa regra é que conseguimos ter arquivos privados dentro dos nossos módulos, como se fossem realmente um pacote separado.

Um exemplo simples na aplicação é o arquivo `~/entities/registration-status.ts`, que exporta algumas classes utilizadas entre os arquivos da pasta `entities`, mas que não devem ser usadas fora dela.

## Prettier

O Prettier também foi configurado para manter o código mais limpo e organizado. A configuração é bem simples, mas o Prettier funciona de forma independente do ESLint.

**Nota:** O ESLint possui um plugin para desativar as regras que conflitam com o Prettier.
